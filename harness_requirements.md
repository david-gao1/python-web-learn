# Harness 风格教程交付要求

## 1. 目标定义要求
- 每个教学单元必须用一句话定义目标，格式为：输入 -> 处理 -> 输出。
- 目标必须可验证，禁止只写“理解某某概念”这类不可验收描述。
- 每个目标需绑定一个业务场景，例如日志处理、规则路由、异步抓取。

## 2. 内容组织要求
- 每节课必须使用固定结构：
  1. 要解决什么问题
  2. 核心思路是什么
  3. 关键机制怎么工作
  4. 怎么实际使用
  5. 思考题（2-3 道）
- 解释顺序必须是“直觉先于细节”，避免先堆定义和术语。
- 代码以“关键片段”展示，不要求从零手写所有底层实现。

## 3. 工程质量要求
- 所有实战代码必须包含最小工程结构：`src/`、`tests/`。
- 必须通过基础质量门禁：
  - `ruff check .`
  - `mypy .`（推荐 `--strict`）
  - `pytest -q`
- 测试应覆盖关键路径和失败路径，避免只测 happy path。

## 4. 可观测与度量要求
- 每个实战任务至少提供 1 个可量化指标：
  - 时延（latency）
  - 吞吐（throughput）
  - 内存占用
  - 错误率
- 指标需要有采集方式和基线对照（如串行 vs 并发）。
- 结果必须可复现，提供运行命令和预期输出示例。

## 5. 验收（Definition of Done）要求
- 功能完成：任务描述中的输入输出行为全部满足。
- 质量通过：lint、类型检查、测试均通过。
- 文档完整：包含运行步骤、关键设计说明、已知边界。
- 复盘到位：提交 2-3 条“本次学到的工程决策”。

## 6. 教程风格要求
- 面向工程落地，避免大段“背景历史”与无关扩展。
- 以真实库和真实场景为主，不鼓励教学型“玩具实现”泛滥。
- 每个概念只解释一次，避免重复“回顾/总结”造成冗余。

## 7. 交付物清单（每周）
- 一份可运行代码（含依赖和运行命令）。
- 一组自动化测试（最少覆盖主链路）。
- 一页内的设计说明（问题、方案、取舍、风险）。
- 一份结果记录（关键指标 + 对比结论）。
- 教学文章：默认 1-3 篇按知识点拆分（每篇 <= 5000 字），可额外提供 1 篇合并长文（可 > 5000 字）。

## 8. 博客发布要求
- 每次学习必须产出教学文章，允许每周 1-3 篇，按知识点拆分发布。
- 每篇必须同时包含概念讲解和实战部分，禁止把“纯概念篇”和“纯实战篇”分开发布。
- 每篇默认字数上限 5000 字（建议区间 2500-4500 字）；允许额外提供合并长文（可 > 5000 字）。
- 每篇必须包含 5 个部分：问题、核心思路、关键机制、实际使用、思考题。
- 每篇必须给出可运行代码片段和对应命令，确保读者可复现。

## 9. 代码内文档要求
- 当引入新的语言特性（装饰器、描述符、数据类、类型重载、协议方法等）时，必须同时提供：
  - 代码中的 docstring 解释：作用、触发方式（语法糖/内建函数如何调用）、工程取舍与边界。
  - 文章中的解释：直觉 + 机制 + 用法 + 常见误区。
- 关于“Python 装饰器/类型注解 vs Java 注解”的统一参考文档：
  - `docs/python_decorators_and_annotations_vs_java.md`
